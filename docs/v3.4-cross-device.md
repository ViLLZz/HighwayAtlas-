# v3.4 Cross-Device Optimization

This document records the 3.4 pass focused on cross-device layout behavior.

## Problem

The atlas had reached a strong visual state on iPad and phone, but some desktop views still felt crowded.

The root cause was that the exporter mainly relied on width breakpoints. That works for many screens, but it does not fully handle cases like:

- wide but relatively short laptop screens
- Mac layouts where floating panels compete with the map
- tablet landscape screens that are wide enough to look like desktop but still behave like touch devices

## 3.4 solution

Instead of shipping separate HTML files for each device, 3.4 makes the atlas choose a layout profile at runtime.

The runtime now classifies the current viewport as:

- desktop
- tablet
- phone
- short-height desktop

Those profiles then change:

- map height
- sidebar vs. stacked layout
- whether map overlays stay floating or become static blocks
- whether the top-of-map quick filters are shown
- floating panel widths and proportions

## Result

The atlas now behaves more like separate tailored versions without actually splitting the product into multiple builds.

That keeps the exporter maintainable while improving the real user experience across:

- Mac and other desktop browsers
- iPad portrait and landscape
- iPhone
- Android phones and tablets

## What changed technically

1. added device-profile detection in the generated runtime
2. added body classes for desktop, tablet, phone, and short-height viewports
3. tuned desktop overlay widths so the map breathes more on large screens
4. made short-height desktop layouts fall back to a cleaner stacked presentation
5. ensured tablet layouts keep the map-first experience even when the screen is wide enough to resemble desktop
6. kept compact-device quick stage filters as part of the profile system instead of relying on one fixed breakpoint

## Why this approach is better than a separate desktop/mobile site

Maintaining separate generated versions would create unnecessary duplication and drift.

The 3.4 profile system keeps:

- one export path
- one data model
- one set of interactions
- one test surface

while still allowing the layout to feel different where it needs to.

## Verification

- exporter smoke tests
